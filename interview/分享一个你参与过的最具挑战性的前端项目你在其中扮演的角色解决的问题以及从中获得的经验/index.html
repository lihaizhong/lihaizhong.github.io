分享一个你参与过的最具挑战性的前端项目，你在其中扮演的角色、解决的问题以及从中获得的经验

查看答案

项目背景

某大型电商平台的 实时库存管理系统，需支持 10万 + SKU 的实时库存展示、多仓调拨和库存预警，用户端包括 PC 管理后台和移动端 APP，日均 PV 50万 +。

我的角色

前端技术负责人，负责架构设计、核心功能开发和性能优化，带领 5 个团队。

挑战与解决方案

1. 实时性与性能矛盾

  • 问题：库存数据每 3 秒更新一次，传统轮询导致频繁请求，页面卡顿。
  • 解决：采用 Web Socket + 增量更新：
    • 后端仅推送变化的库存数据（而非全量），减少数据传输量。
    • 前端用 requestAnimationFrame 批量更新 DOM，避免频繁回流。

2. 复杂表格交互

  • 问题：10万+ 行数据的表格需支持排序、筛选、冻结列，传统表格组件性能崩溃。
  • 解决：实现 虚拟滚动表格：
    • 仅渲染可视区域内的行（约 50 行），滚动时动态计算渲染范围。
    • 基于 IndexDB 缓存筛选后的全局数据，避免重复请求。

3. 多端适配与一致性

  • 问题：PC 端（复杂表单）和移动端（简化操作）需共享业务逻辑，但 UI 差异大。
  • 解决：采用 逻辑与 UI 分离 架构：
    • 核心逻辑（如库存计算、校验规则）封装为独立 SDK，多端共享。
    • UI 层使用不同组件库（PC 用 AntDesign，移动端用 Vant），通过适配层统一接口。

经验总结

  • 技术选型需权衡：实时场景优先考虑 WebSocket 而非轮询，但需做好断线重连和降级方案。
  • 性能优化要量化：通过 Lighthouse 和 Performance API 建立性能基准，针对性优化（如首屏加载从 3s 降到 1.2s）。
  • 团队协作靠规范：制定组件设计规范和 API 文档标准，减少沟通成本（如统一用 Swagger 管理接口文档）。
  • 故障预案不可少：针对 WebSocket 连接失败、数据同步异常等场景，设计降级方案（如自动切换为轮询）。
